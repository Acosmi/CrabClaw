# 渠道运维

用于飞书/企微/钉钉等渠道的状态检查、配置保存、文本发送验证与收件链路回归。

本技能只覆盖渠道连通与文本投递主路径。  
如果要把截图、PDF、代码文件发到远程频道，切换到 `send-media`，不要继续用 `send` 硬做文件发送。

## 覆盖方法
- `channels.status`
- `channels.save`
- `channels.logout`
- `send`
- `poll`

## 何时使用

- 新接入一个远程频道，要先确认配置与凭据状态
- 需要验证文本消息能否送达目标群/用户
- 需要确认轮询型渠道是否能正常收件
- 需要为后续的远程审批通知、主动执行结果投递先打通渠道主链路

## 与其他技能的边界

- `send`：文本消息发送
- `send-media`：文件、图片、文档发送
- `agent-send`：无入站消息时主动跑一次 Agent 并可投递最终回复
- `cron-ops`：定时触发并投递结果

## 推荐流程
1. 先查 `channels.status`，识别已配置/未配置渠道。
2. 用 `channels.save` 写入最小配置并再次读取状态。
3. 用 `send` 发一条短文本，先验证目标路由、错误码和回执。
4. 如果业务要发文件或截图，改用 `send_media` 做第二次验收，不要把 `send` 当成文件通道。
5. 涉及轮询渠道时用 `poll` 验证收件能力。
6. 失效凭据用 `channels.logout` 清理后重配。

## 工作原则

- 先打通文本链路，再验文件链路。
- 已在远程会话上下文内时，优先沿当前会话路由发送；不要臆造目标 ID。
- 对无人值守的主动任务，渠道和目标最好显式写清，不要长期依赖模糊的“上次路由”。

## 故障树

- `channels.save` 成功但渠道仍不可用：先区分是首次配置需要 Gateway 重启，还是凭证本身已失效
- `send` 成功但发错地方：通常是 `to`、渠道别名或会话路由错了，不是发送链路本身坏了
- 文本能发、文件不能发：这是文件链路未单独验收，应切到 `send-media` 排查，不要继续用 `send`
- `poll` 失败：先确认目标渠道是否支持 poll，再确认不是只测了出站没测收件
- `channels.status` 正常但业务仍失败：通常是“已配置”不等于“目标路由正确”或“凭证未过期”

## 回滚步骤

- 发现凭证或配置可疑时，先用 `channels.logout` 清掉失效状态，再重新 `channels.save`
- 首次配置后若返回需要重启，先完成重启再继续发消息验收
- 文本链路未通前，不要继续测文件链路、审批通知或主动投递
- 目标路由不确定时，退回到一条最短文本消息重新验证，不要直接发业务消息
- `poll` 不稳定时，先把出站链路和目标插件支持性确认清楚，再决定是否继续做收件回归

## 验收清单

- `channels.status` 显示目标渠道已配置，且运行状态与预期一致
- `send` 的短文本已送达正确目标，而不是只返回接口成功
- 若后续流程需要文件或截图，`send_media` 已单独验收
- 若后续流程依赖收件，`poll` 或等效收件链路已实际验证
- 整个渠道验收没有把“文本成功”误当成“所有交付能力都成功”
