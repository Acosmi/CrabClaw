# send_media

`send_media` 用于把本地文件或内存中的媒体数据投递到远程频道。

## 何时使用

- 用户明确要求“把这个文件发过去”
- 已经在本地生成了 PDF、PPT、截图、代码文件，需要投递到飞书等远程频道
- 需要把截图或图表作为文件/图片发出，而不是只发文字描述

典型流程：

1. 先在本地生成或确认文件
2. 再调用 `send_media`
3. 需要说明文字时，放进 `message`

不要用 `send` 伪装文件发送；`send` 只适合文本。

## 输入要点

- `file_path`：优先使用。必须是绝对路径，且路径需落在工具可访问范围内。
- `file_name`：在 `media_base64` 模式下建议显式提供，用于保留远程附件文件名。
- `media_base64`：只在数据已经在内存里时使用。
- `target`：可选，格式为 `channel:id`。省略时会沿当前会话路由发送。
- `mime_type`：通常可省略；运行时会按扩展名自动推断。
- `message`：文件附带说明文字。

## 当前实现边界

- 大小限制为 30MB。
- 使用 `file_path` 时，会自动沿用本地 basename 作为远程附件文件名。
- 未提供 `target` 时，会回落到当前 `sessionKey`；如果当前会话没有远程渠道路由，会直接失败。
- 不要臆造频道 ID；没有现成路由时，明确传 `target`。
- MIME 会自动识别常见图片、PDF 和 Office 文档。
- 纯 Web 模式或未配置频道时，工具可能不可用。

## 注意事项

- `file_path` 优先于 `media_base64`，避免无谓的 base64 膨胀。
- 发送前先确认文件真实存在，且路径在工作区或授权范围内。
- 需要“主动跑任务再发结果”时，通常是 `agent-send`/`cron-ops` 负责生成结果，`send_media` 负责文件投递。

## 故障树

- 工具直接报 `Media sender is not available`：说明当前运行上下文没有可用频道发送器，不是文件本身坏了
- 报 `No target specified and no session channel available`：说明既没有当前远程会话路由，也没有显式 `target`
- 报路径访问或文件不存在：先看 `file_path` 是否绝对路径、是否在可访问范围内
- 报超过 30MB：这是实现硬限制，应先压缩或改交付方式
- 发送阶段失败：优先看错误里的 `sendCode`，区分未授权、上游不可用、通道不支持该媒体能力

## 回滚步骤

- 先停止重复发送，避免对同一目标反复投大文件
- 有路径问题时先本地确认文件存在，再重发；不要直接切回 `media_base64`
- 超过 30MB 时先压缩、分包或改成文本说明加链接
- 没有远程路由时，显式补 `target`；不要编造会话 ID
- 通道明确不支持该媒体类型时，退回文本说明或可公开访问的链接

## 验收清单

- 返回结果中的 `target` 与预期渠道一致
- 目标端实际收到文件或图片，而不是只看到接口返回
- 发送用的是正确输入源：优先 `file_path`，不是无谓的 base64
- 文件大小、MIME、渠道能力都与当前实现边界一致
- 若失败，已能从错误信息判断是路径、路由、大小还是通道能力问题
