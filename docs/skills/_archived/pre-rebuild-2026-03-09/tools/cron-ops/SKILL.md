# Cron 运维

用于定时任务新增、更新、立即执行、运行历史追踪，以及无人值守投递链路验收。

## 覆盖方法
- `wake`
- `cron.list`
- `cron.status`
- `cron.runs`
- `cron.add`
- `cron.update`
- `cron.remove`
- `cron.run`

## 何时使用

- 需要定时或周期性执行任务
- 需要无人值守地生成结果并投递到渠道
- 需要对已有 cron 任务做回归、补跑或排障

## 与其他能力的边界

- `agent-send`：立刻主动跑一次，不负责长期调度
- `hooks-agent`：由外部事件触发，不是固定时间调度
- `send-media`：当 cron 任务产出的是文件时，文件交付仍需单独设计和验收

## 标准流程
1. 用 `cron.list` 获取现有任务与冲突项。
2. 用 `cron.add` 创建最小任务，再用 `cron.status` 确认调度器健康。
3. 涉及远程投递时，优先显式设置 delivery 的 `channel` 与 `to`，不要长期依赖模糊的 `last`。
4. 用 `cron.run` 人工触发一次做验收，先看能不能跑，再看能不能送达。
5. 用 `cron.runs` 检查最近执行耗时、错误、会话关联和投递结果。
6. 变更时走 `cron.update`，废弃任务走 `cron.remove`。

## 变更原则
- 先低频后高频，避免一次性高并发调度。
- 任务名、用途、输出路径保持可审计。
- 定时任务一旦需要对外发消息，目标渠道和接收方尽量显式化。
- `wake` 更适合唤醒已有心跳/队列，不替代完整的 cron 调度设计。

## 故障树

- `cron.status` 正常但业务没产出：调度器活着不等于任务本身配置正确，先看 `cron.runs`
- `cron.run` 能跑但无人值守时没送达：通常是 delivery 目标缺失，或长期依赖了模糊的 `last`
- 任务成功但外部没有消息：要区分 `delivery.mode` 是 `none`，还是目标 `channel/to` 缺失或无效
- 产物是文件却只发了文本摘要：说明文件交付设计缺失，应补 `send_media`
- 周期任务连续报错：先看任务内容与 delivery 配置，再看重试回退，不要只盯调度器健康

## 回滚步骤

- 先用 `cron.run` 手动补跑一遍，确认任务逻辑本身能成功
- delivery 不稳定时，先显式写 `channel` 与 `to`，停止依赖长期的 `last`
- 周期设得过激时，先降频，再继续验收投递链路
- 任务不再需要对外发送时，临时改成内部输出，避免持续误投
- 文件型产物先保留内部运行，再把文件交付拆给 `send_media`

## 验收清单

- `cron.status` 正常，且最近运行记录可追踪
- `cron.runs` 中有成功记录，摘要/错误信息足以排障
- 若要求对外投递，目标渠道已实际收到结果
- 若任务产出文件，文件交付链路已单独设计并验收
- 整个任务在无人值守条件下仍能稳定复现，不依赖人工补路由
