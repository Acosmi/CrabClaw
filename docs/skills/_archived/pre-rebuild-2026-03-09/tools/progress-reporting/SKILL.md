# 中间进度汇报

`report_progress` 用于在长任务中主动汇报阶段性进展。  
它是**中途状态信号**，不是最终结果；远程聊天渠道只有在入口显式接线时才会收到节流后的阶段性文本。

## 何时使用

- 任务明显分成多个阶段
- 已完成一个关键步骤，准备进入下一阶段
- 任务因为审批、构建、测试、等待外部资源而变慢

短任务、单步任务、几秒内完成的任务一般不要调用。

## 输入规则

- `summary`：必填，简短说明当前进展
- `percent`：可选，只有 `1-100` 会被保留
- `phase`：可选，如 `analyzing`、`implementing`、`testing`

运行时会把过长的 `summary` 截断到约 300 字。

## 推荐粒度

- 每完成一个关键阶段报一次
- 卡住或等待时补一次
- 不要按“每个命令”“每张截图”“每次重试”高频刷进度

好的例子：

- “已完成代码扫描，开始修改权限路径。”
- “构建完成，正在跑回归测试。”
- “已发起审批，等待用户确认。”

## 当前实现边界

- 运行时会发出 `agent.progress` 事件。
- 已确认的实时消费路径是 WebSocket / 任务面板类实时界面。
- 当前部分显式接线的远程入站渠道会把进度镜像成粗粒度文本更新，并做节流/去重。
- 当前标准 OpenAI/OpenResponses SSE 主要只转发 `assistant` 与 `lifecycle`，不要假设它们会稳定透传进度事件。
- `webchat`、OpenAI compat、OpenResponses、agent RPC 这类入口仍应按 UI-only 理解，不要假设它们会把中途进度外发。

## 使用原则

- 用它告诉用户“任务还在继续，而且到了哪一步”。
- 不要把它当成最终答复；最终结果仍要正常回复或正常 `deliver`。
- 需要远程用户看到中途状态时，只在入口已显式接线的前提下使用它；否则仍要拆成独立投递。
- 如果最终产物是文件，文件交付仍走 `send_media`。

## 故障树

- `summary` 为空：运行时会直接拒绝，这不是展示问题而是输入不合法
- 百分比看起来很假：通常是没有阶段划分，导致 `percent` 成了拍脑袋数字
- 高频刷进度：说明把工具当日志流用了，而不是阶段性汇报
- 远程频道没看到进度：先区分入口是否已显式接线，再看是否命中了节流/去重
- 只有进度没有最终结果：说明把 `report_progress` 错当成最终回复链路了

## 回滚步骤

- 进度内容含糊时，先停下来把阶段重新压缩成一句可信摘要
- 如果发现自己在每个命令后都想汇报，立即降频，只保留关键阶段
- 远程入口未接线时，停止依赖 `report_progress` 做外发，改成显式发送或多次独立投递
- 连续两次更新太密时，先等待阶段真的推进，再发下一条，避免一直命中节流
- 若已经发了进度但任务结束，仍要正常输出最终答复或最终交付
- 百分比无法可信估计时，宁可不填，也不要硬报

## 验收清单

- 支持实时事件的界面能看到阶段性更新
- 已接线的远程渠道只收到粗粒度、非刷屏的阶段性文本
- 每条进度都有明确阶段意义，而不是命令流水账
- `summary`、`phase`、`percent` 与任务真实状态一致
- 进度之后仍有正常的最终答复或最终 deliver
- 没有把 `report_progress` 误写成“所有入口都稳定外发”的能力
